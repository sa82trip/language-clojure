지난 글에서는 `let` 으로 이름을 붙이고 `if` 와 `when` , `cond` 로 갈래를 나눴다. 이제 마지막 한 가지, `반복` 이 남았다. 3편에서 `map` 과 `filter` , `reduce` 로 컬렉션을 바꾸는 법을 배웠다 — 대부분의 반복은 저 셋으로 충분하다. 하지만 값을 누적하며 조건이 바뀌는 루프, 혹은 `reduce` 로 표현하면 오히려 어색한 순간에는 `명시적 반복` 이 필요하다.

클로저엔 C의 `for` 문이나 `while` 문이 없다(이름 비슷한 건 있는데, 아래에서 보자). 대신 `recur` 와 `loop` 가 있다. 이번 글에서는 일반 재귀의 함정부터 시작해 `recur` 로 빠져나가는 법, `loop` 로 루프를 짜는 법, 그리고 부수효과 반복까지 본다.

아래 코드는 전부 `clj` 로 직접 실행한 진짜 결과다.


## 일반 재귀부터 — 친숙하지만 함정 {#일반-재귀부터-친숙하지만-함정}

팩토리얼은 반복의 단골 예제. 먼저 익숙한 모양으로, 함수가 자기 자신을 부르는 `일반 재귀` 로 짜보자.

```clojure
(defn fact [n]
  (if (zero? n) 1
    (*' n (fact (dec n)))))     ; n * (n-1)!

(fact 5)    ;; => 120
(fact 10)   ;; => 3628800
(fact 20)   ;; => 2432902008176640000
```

작은 수는 잘 된다. 그런데 `(fact 5)` 를 풀면 `5 * (fact 4)` 이고, `(fact 4)` 는 다시 `4 * (fact 3)` ... 이렇게 내려간 뒤, `(fact 0)` 이 `1` 이 되어서야 거꾸로 올라오며 곱한다. `곱하기를 미뤄둔 채` 로 `호출이 스택에 쌓인다`.

참고로, 클로저의 `*` 은 `long` 오버플로를 검사해서 예외를 던지고, `*'` 는 자동으로 큰 정수(`BigInteger`)로 승격한다. 팩토리얼처럼 금방 커지는 계산엔 `*'` 를 쓴다.

스택이 쌓인다는 게 왜 문제일까? 숫자를 키워보자.

```clojure
(fact 100000)
;; => StackOverflowError
```

JVM은 함수 호출을 스택에 쌓고, 재귀가 너무 깊어지면 스택이 바닥난다. `map` 과 `reduce` 가 게으르고 스택을 안 쌓는 것과 대조적이다. 클로저가 재귀를 막는 건 아니다 — 다만 `무한히 깊어질 수 있는 재귀` 에는 별도 도구가 필요하다.


## recur — 꼬리에서 다시 부르기 {#recur-꼬리에서-다시-부르기}

함수형 언어는 `꼬리 호출 최적화(TCO)` 로 이 문제를 푼다: 재귀 호출이 함수의 `맨 마지막 동작(꼬리)` 이면, 새 호출을 스택에 쌓지 않고 `현재 프레임을 재활용` 한다. JVM은 이 최적화를 언어 차원에서 지원하지 않는다. 그래서 클로저는 `recur` 라는 특수 폼으로, `이 호출은 꼬리재귀다, 스택을 새로 쓰지 마` 라고 `명시` 한다.

꼬리재귀로 팩토리얼을 다시 쓰려면, `아직 곱하지 않은 값` 을 `함께 들고 다녀야` 한다. 이 `누적값` 을 `acc` 라고 부르자. 그리고 `loop` 가 그 바인딩을 잡아준다.


## loop/recur — 명시적 루프 {#loop-recur-명시적-루프}

`loop` 는 `let` 과 닮았다: 이름에 값을 바인딩한다. 차이는 `recur` 로 `그 바인딩을 새 값으로 다시 시작` 할 수 있다는 것. 매번 새 스택 프레임을 만들지 않고 `같은 프레임을 재활용` 한다.

```clojure
(defn fact [n]
  (loop [n n        ; 카운터 (시작값 = n)
         acc 1]     ; 누적값 (시작값 = 1)
    (if (zero? n) acc
      (recur (dec n) (*' acc n)))))   ; n-1, acc*n 으로 재시작

(fact 5)    ;; => 120
(fact 10)   ;; => 3628800
(fact 20)   ;; => 2432902008176640000
```

읽는 법:

-   `loop` 의 `[n n acc 1]` 은 `let` 처럼 `n` 에 `n` , `acc` 에 `1` 을 바인딩.
-   `(zero? n)` 이 참이면 `acc` 를 반환(루프 종료).
-   아니면 `recur` 로 `n` 은 `1 줄이고` , `acc` 는 `현재 n을 곱해` 다시 시작. `recur` 의 인자 순서는 `loop` 의 바인딩 순서와 `정확히` 맞아야 한다.

핵심은 `누적값(accumulator)` 패턴이다. 일반 재귀에선 `나중에 곱할` 값을 스택에 미뤄뒀지만, 꼬리재귀에선 `지금 곱한 결과` 를 `acc` 에 들고 다닌다. 그래야 `recur` 가 `맨 마지막 동작` 이 되어 스택이 안 쌓인다.

큰 수를 줘도 이번엔 안전하다:

```clojure
(fact 10000)
;; => 2846259680917054518906413... (총 35660자리)
```

스택은 그대로고, 그냥 `n` 이 `0` 이 될 때까지 `acc` 를 갱신할 뿐이다.


## recur는 꼬리에서만 {#recur는-꼬리에서만}

`recur` 는 `반드시 꼬리 위치` 에 있어야 한다. `recur` 의 결과를 `다시 다른 식에 쓰면` 안 된다 — 그러면 `recur` 가 끝이 아니라 `중간 단계` 가 되어 스택을 써야 하기 때문이다.

```clojure
(defn bad [n]
  (if (zero? n) 0
    (+ 1 (recur (dec n)))))   ; recur 결과에 +1 → 꼬리 아님
;; => 컴파일 에러: Can only recur from tail position
```

`recur` 가 `+` 의 인자로 들어갔다. `recur` 결과에 `1` 을 더해야 하니, `recur` 는 `끝` 이 아니다. 클로저 컴파일러가 이걸 `컴파일 타임` 에 잡아낸다 — 런타임에 스택이 터지는 대신. 이게 `recur` 의 안전장치다.


## 다른 예 — 누적 합과 카운트다운 {#다른-예-누적-합과-카운트다운}

같은 패턴이면 뭐든 된다. `1` 부터 `n` 까지 합:

```clojure
(defn sum-to [n]
  (loop [i 1
         acc 0]
    (if (> i n) acc
      (recur (inc i) (+ acc i)))))

(sum-to 10)    ;; => 55
(sum-to 100)   ;; => 5050
```

카운트다운처럼 `컬렉션을 만들며` 누적할 수도 있다:

```clojure
(defn countdown [n]
  (loop [i n
         acc []]
    (if (neg? i) acc
      (recur (dec i) (conj acc i)))))   ; acc에 i를 추가

(countdown 5)
;; => [5 4 3 2 1 0]
```

`conj` 로 `acc` (벡터)에 `i` 를 붙였다. `loop` 와 `recur` 는 단순히 숫자만 누적하는 게 아니라 `상태` 를 들고 가는 `작은 기계` 처럼 쓸 수 있다.


## doseq / dotimes — 부수효과 반복 {#doseq-dotimes-부수효과-반복}

`loop` 와 `recur` 가 `값` 을 만드는 반복이라면, `그냥 무언가를 할` 때(출력, 로그, DB 저장)는 `doseq` 와 `dotimes` 가 더 직결이다. 둘 다 `nil` 을 반환한다 — 값이 목적이 아니기 때문이다.

```clojure
(doseq [x [1 2 3]]
  (println x))
;; 1
;; 2
;; 3
;; => nil

(dotimes [i 3]
  (println "i:" i))
;; i: 0
;; i: 1
;; i: 2
;; => nil
```

`doseq` 는 컬렉션 원소마다, `dotimes` 는 정해진 횟수만큼. 둘 다 `for` 문이 아니라 `부수효과용` 이다. 반환값을 쓸 일이 없을 때만 쓴다.


## for — 값을 만드는 반복 (리스트 컴프리헨션) {#for-값을-만드는-반복--리스트-컴프리헨션}

`for` 라는 이름이 나왔다. 하지만 클로저의 `for` 는 C의 `for` 문이 아니라 `리스트 컴프리헨션` 이다 — 여러 컬렉션을 `조합해 새 리스트` 를 만든다.

```clojure
(for [x [1 2 3]
      y [:a :b]]
  [x y])
;; => ([1 :a] [1 :b] [2 :a] [2 :b] [3 :a] [3 :b])
```

`x` 와 `y` 의 모든 조합. `:when` 으로 거를 수도 있다:

```clojure
(for [x (range 10)
      :when (even? x)]
  x)
;; => (0 2 4 6 8)
```

`for` 는 `값` 을 만들고 `doseq` 는 `부수효과` 를 낸다 — 형태는 거의 같지만 목적이 다르다.


## 그럼 reduce는? — 언제 loop/recur {#그럼-reduce는-언제-loop-recur}

잠깐, 3편에서 `reduce` 로 뭘든 합칠 수 있다고 했다. 팩토리얼도 `reduce` 한 줄로 된다:

```clojure
(defn fact [n]
  (reduce *' 1 (range 1 (inc n))))

(fact 5)    ;; => 120
```

그렇다면 `loop` 와 `recur` 는 언제? `reduce` 는 `하나의 컬렉션을 하나의 값으로 줄일 때` 딱 맞다. 하지만:

-   `상태가 여러 개` 며 `매 단계마다 다 바뀔 때` (카운터 + 누적 벡터 + 플래그 등),
-   종료 조건이 단순히 컬렉션 끝이 아닐 때 (어떤 기준을 넘으면 멈춘다거나),
-   입력이 컬렉션이 아니거나, `여러 컬렉션을 엇갈리며` 읽을 때,

`loop` 와 `recur` 가 더 자연스럽다. `reduce` 로 억지로 끼워맞추는 것보다.


## 실전 — 문자 빈도수 세기 {#실전-문자-빈도수-세기}

문자열에서 글자별 등장 횟수를 세보자. `loop` 와 `recur` , 맵 누적의 전형적인 예다:

```clojure
(defn freq [s]
  (loop [s (seq s)        ; 시퀀스로 변환 (5편 seq 관용구)
         acc {}]          ; 빈 맵에서 시작
    (if (empty? s) acc
      (let [c (first s)
            acc (assoc acc c (inc (get acc c 0)))]   ; c의 카운트 +1
        (recur (rest s) acc)))))

(freq "hello")
;; => {\h 1, \e 1, \l 2, \o 1}
```

읽는 법:

-   `(seq s)` 로 문자열을 한 글자씩의 시퀀스로 (5편의 `seq` 관용구).
-   `acc` 는 `글자 → 개수` 맵. `(get acc c 0)` 으로 없으면 `0` , 있으면 현재 개수.
-   `(inc ...)` 해서 `+1` 한 뒤 `assoc` 으로 `acc` 를 갱신.
-   `(rest s)` 로 다음 글자로 넘어가며 `recur` .

물론 이건 `(frequencies "hello")` 한 줄로도 된다 — 클로저에 이미 있다. 하지만 `loop` 와 `recur` 로 직접 짜보면 `누적값 패턴` 이 손에 익는다.


## 언제 뭘 쓸까 {#언제-뭘-쓸까}

-   컬렉션을 `변환/필터/집계` : `map` 과 `filter` , `reduce` (3편). 대부분은 이걸로 끝.
-   `값을 누적하며 루프` , 종료 조건이 단순치 않을 때: `loop` 와 `recur` .
-   `부수효과` (출력·저장): `doseq` 와 `dotimes` . `nil` 반환.
-   `조합된 값 만들기` (데카르트 곱 등): `for` .
-   `재귀적 데이터 구조` (트리 순회): `recur` 로 함수 자기 호출.
-   일반 재귀( `recur` 없는)는 `작고 얕을 때만` . 깊어지면 스택이 터진다.


## 10분 요약 {#10분-요약}

-   일반 재귀는 직관적이지만 `호출 스택` 을 쌓는다. 깊어지면 `StackOverflowError` .
-   `recur` : `꼬리 위치` 에서 자기 자신(또는 `loop` )을 다시 부르는 특수 폼. 스택을 `재활용` 한다.
-   `loop` 와 `recur` : `let` 처럼 바인딩을 주고, `recur` 로 `새 값으로 재시작` . `누적값(acc)` 을 들고 다니는 게 핵심.
-   `recur` 는 `꼬리에서만` . 결과를 다른 식에 쓰면 `컴파일 에러` ( `Can only recur from tail position` ).
-   `doseq` 와 `dotimes` : 부수효과용. `nil` 반환.
-   `for` : 리스트 컴프리헨션. `값` 을 만든다( `:when` 가드 가능).
-   `reduce` 로 끝나면 `reduce` 쓰고; `여러 상태` 나 `복잡한 종료 조건` 이면 `loop` 와 `recur` .

다음 글에서는 클로저의 `가장 큰 특징` 중 하나, `불변` 을 다룬다. 지금까지 본 모든 데이터는 `한 번 만들면 안 바뀌었다` . 하지만 실전에선 카운터, 캐시, 설정처럼 `시간에 따라 변하는 값` 이 필요하다. 클로저는 `atom` 으로 `안전하게` 상태를 바꾼다 — 불변을 깨지 않으면서.
