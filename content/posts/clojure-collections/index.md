설치를 끝냈으니, 이제 진짜 클로저를 만져봅시다. 클로저를 처음 배울 때 가장 먼저 익혀야 할 건 `함수` 가 아니라 `데이터` 다. 클로저는 데이터 중심 언어라, 컬렉션을 다루는 방법이 곧 클로저 사고방식의 출발점이다.

클로저에는 4가지 핵심 컬렉션이 있다. 리터럴(코드에 직접 적는 표기)만 봐도 각각의 성질이 드러난다.

| 컬렉션 | 리터럴     | 주요 특징        |
|-----|---------|--------------|
| 리스트 | `(1 2 3)`  | 머리에 추가, 순차 접근 |
| 벡터 | `[1 2 3]`  | 인덱스 접근, 꼬리에 추가 |
| 맵  | `{:a 1}`   | 키-값 쌍, 키워드가 키 역할 |
| 셋  | `#{1 2 3}` | 중복 불가, 유일 원소 모음 |


## 왜 컬렉션부터인가 {#왜-컬렉션부터인가}

대부분의 언어는 변수·조건문·반복문부터 가르친다. 클로저는 다르다. 반복은 `재귀` 나 `시퀀스 추상화` 로, 조건은 `if` 로 처리하지만, 그 모든 게 결국 `컬렉션을 변환` 하는 일로 귀결된다. 그러니 컬렉션 리터럴과 기본 함수부터 손에 익히는 게 가장 빠른 길이다.

아래 코드는 `clj` REPL 이나 스크립트에서 직접 실행해 본 결과다. 거짓 없는 진짜 출력이다.


## 리스트 (list) — 링크드 리스트 {#리스트--list--링크드-리스트}

가장 클로저스러운(리스프 뿌리) 자료구조. 원소를 `앞` 에 붙인다.

```clojure
'(1 2 3)                       ; 리스트 리터럴
(list 1 2 3)                   ; list 함수
(first '(1 2 3))               ; => 1
(rest '(1 2 3))                ; => (2 3)
(conj '(1 2 3) 0)              ; => (0 1 2 3)  -- 앞에 추가
(count '(1 2 3))               ; => 3
```

```text
(first ...) => 1
(rest ...)  => (2 3)
(conj ...)  => (0 1 2 3)
(count ...) => 3
```


## 벡터 (vector) — 인덱스 접근이 빠른 배열 {#벡터--vector--인덱스-접근이-빠른-배열}

대부분의 경우 리스트보다 벡터를 더 자주 쓴다. 인덱스로 바로 찾아간다.

```clojure
[1 2 3]                        ; 벡터 리터럴
(get [10 20 30] 1)             ; => 20  (get 함수)
([10 20 30] 1)                 ; => 20  (벡터 자체를 함수처럼!)
(conj [1 2 3] 4)               ; => [1 2 3 4]  -- 뒤에 추가
(assoc [1 2 3] 0 99)           ; => [99 2 3]   -- 인덱스 0을 99로
```

```text
(get ...)  => 20
([..] 1)   => 20
(conj ...) => [1 2 3 4]
(assoc ...) => [99 2 3]
```

재미있는 점: 클로저에서는 `벡터 자체를 함수처럼` 호출할 수 있다. `([10 20 30] 1)` 은 "벡터의 인덱스 1 원소를 가져와라"는 뜻. 데이터가 곧 함수처럼 동작하는, 클로저의 매력 중 하나.


## 맵 (map) — 키-값 사전 {#맵--map--키-값-사전}

가장 많이 쓰는 컬렉션. 키에는 보통 `키워드` (= :a :name := 같이 콜론으로 시작) 를 쓴다.

```clojure
{:name "정탐" :lang "clojure"}            ; 맵 리터럴
(:name {:name "정탐" :lang "clojure"})    ; => "정탐"  (키워드가 함수!)
(get {:name "정탐" :lang "clojure"} :lang) ; => "clojure"
(keys {:a 1 :b 2 :c 3})                    ; => (:a :b :c)
(vals {:a 1 :b 2 :c 3})                    ; => (1 2 3)
(assoc {:a 1} :b 2)                        ; => {:a 1, :b 2}   항목 추가
(dissoc {:a 1 :b 2} :a)                    ; => {:b 2}         항목 제거
```

```text
(:name ...) => 정탐
(get ...)   => clojure
(keys ...)  => (:a :b :c)
(vals ...)  => (1 2 3)
(assoc ...) => {:a 1, :b 2}
(dissoc ...) => {:b 2}
```

벡터와 마찬가지로 `키워드가 함수` 가 된다. `(:name 맵)` 처럼 쓴다. 이 패턴이 클로저 코드에서 정말 많이 등장한다.


## 셋 (set) — 유일 원소의 모임 {#셋--set--유일-원소의-모임}

중복을 자동으로 없앤다. 포함 여부 검사에 좋다.

```clojure
#{1 2 3}                       ; 셋 리터럴
(set [1 1 2 2 3 3])            ; => #{1 3 2}  (중복 제거)
(contains? #{1 2 3} 2)         ; => true
(conj #{1 2 3} 4)              ; => #{1 4 3 2}
```

```text
(set ...)       => #{1 3 2}
(contains? ...) => true
(conj ...)      => #{1 4 3 2}
```


## 핵심 철학 — 원본은 변하지 않는다 (불변성) {#핵심-철학-원본은-변하지-않는다--불변성}

지금까지 `conj` , `assoc` , `dissoc` 를 썼지만, `한 가지` 를 눈치챘을 것이다. 어느 것도 `원본을 고치지 않았다`. 클로저 컬렉션은 `불변(immutable)` 이다.

```clojure
(def 원본 [1 2 3])
(def 변경 (conj 원본 4))

원본                           ; => [1 2 3]     여전히 원본!
변경                           ; => [1 2 3 4]   새로 만들어진 값
```

```text
원본 => [1 2 3]
변경 => [1 2 3 4]
원본 => [1 2 3]   ; conj 후에도 원본은 안 변함
```

`conj` 는 원본을 고치는 게 아니라 `새로운 컬렉션` 을 돌려준다. 그렇다고 매번 전체를 복사해서 비싼 건 아니다 — 클로저는 `영속 데이터 구조(persistent data structure)` 를 써서 변경된 부분만 새로 만들고 나머지는 공유한다. "복사 + 공유"의 마법.

이 불변성이 클로저가 동시성 문제에서 강력한 이유 중 하나다. 여러 곳에서 같은 컬렉션을 읽어도 아무도 못 바꾸니까 락을 덜 쓴다.


## 10분 요약 {#10분-요약}

-   4가지 컬렉션 리터럴: `()` 리스트, `[]` 벡터, `{:k v}` 맵, `#{}` 셋.
-   공통 함수: `count` , `conj` , `first` , `rest` .
-   맵 전용: `get` , `keys` , `vals` , `assoc` , `dissoc` .
-   벡터와 키워드는 `함수처럼` 호출할 수 있다.
-   모든 변경 함수는 `원본을 두고 새 값을 돌려준다` (불변 + 영속 구조).

다음 글에서는 이 컬렉션들을 한 번에 순회하고 변환하는 클로저의 핵심 무기, `시퀀스 추상화` 와 `map` / `filter` / `reduce` 를 다룬다. 그게 진짜 클로저의 힘이 보이는 순간이다.
